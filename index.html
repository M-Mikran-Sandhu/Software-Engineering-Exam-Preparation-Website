Section 1: Course Introduction and Fundamentals

1

1.1 Software Engineering Introduction

Software Engineering represents a systematic approach to the design, development, and maintenance of software systems. It integrates principles from computer science, project management, and engineering disciplines to create reliable, efficient, and maintainable software solutions.

Core Definitions:

Program: A sequence of instructions written in a programming language that can be executed by a computer to perform specific tasks.

Software: All non-physical components of a computing system, including programs, data, documentation, and configuration files.

Engineering: The practical application of scientific and mathematical principles to design and build structures, machines, or systems.

Software Engineering (SE): The disciplined application of engineering principles to software development, encompassing the entire lifecycle from requirements to maintenance.

Computer Science (CS): Focuses on theoretical foundations of computation, algorithms, data structures, and computational complexity.

1.2 Software Engineering Terminology

Method: A systematic procedure or technique for accomplishing a task.

Tool: Software or hardware that supports the execution of methods.

Procedure: A step-by-step description of how to perform a task or apply a method.

Paradigm: A fundamental style or approach to programming and problem-solving.

1.3 Error, Fault, and Failure

Error: A human action that results in incorrect code or logic.

Fault: The manifestation of an error in the code; a defect in the software.

Failure: Occurs when a fault is executed and causes the software to behave incorrectly.

Relationship: Error → Fault → Failure

1.4 Systems Approach

A System is an integrated set of elements designed to accomplish a defined objective. Systems consist of Objects (entities with properties and state) and Activities (processes and operations).

Relationships between Objects and Activities: Objects are acted upon by activities, and activities operate on objects to transform their state.

Interrelated Systems and Sub-systems: Complex systems are often composed of interconnected sub-systems that can be analyzed independently while understanding their interactions.

Section 2: Engineering Approach and Process Models

2

2.1 Phases of Software Engineering

The software engineering lifecycle typically includes these phases:

  • Requirements Analysis: Understanding what the software must do and documenting specifications.
  • Design: Creating the architecture and detailed design of the system.
  • Implementation: Writing code according to design specifications.
  • Testing: Verifying that the software meets requirements and identifying defects.
  • Deployment: Releasing the software to production environments.
  • Maintenance: Providing ongoing support, fixing bugs, and implementing enhancements.

2.2 Development Team Members and Roles

Project Manager: Oversees project schedule, budget, and resource allocation.

Business Analyst: Gathers requirements from stakeholders and documents specifications.

Software Architect: Designs overall system structure and makes technology decisions.

Software Developer: Implements code based on design specifications.

Quality Assurance Engineer: Develops test plans and executes tests.

DevOps Engineer: Manages infrastructure and deployment pipelines.

2.3 Software Process Models

Waterfall Model: A linear, sequential approach where each phase must be completed before the next begins. Advantages: clear structure and documentation. Disadvantages: inflexibility and late defect detection.

Waterfall with Prototyping: Combines waterfall's structure with prototyping to validate requirements early.

V Model: Extends waterfall by emphasizing testing at each development phase, creating a V-shaped diagram.

Prototyping Model: Emphasizes building prototypes to explore requirements and design alternatives.

Section 3: Advanced Process Models and Project Management

3

3.1 Additional Process Models

Phased Development: Divides the project into multiple phases, each delivering a subset of functionality.

Spiral Model: Combines iterative development with risk management. Each spiral iteration includes planning, risk analysis, development, and evaluation.

Unified Process (UP): A comprehensive framework emphasizing iterative development and use case-driven requirements.

Agile (Scrum): An iterative, incremental approach emphasizing flexibility and customer collaboration.

Extreme Programming (XP): An agile methodology emphasizing technical excellence through pair programming and test-driven development.

3.2 Project Management Fundamentals

Work Breakdown Structure (WBS): A hierarchical decomposition of the project into manageable phases, steps, and activities.

Project Schedule: A timeline showing when each activity will occur, considering dependencies and resource constraints.

Activity Graphs: Visual representations of project activities and their dependencies.

Critical Path Method (CPM): An algorithm for identifying the longest sequence of dependent activities.

Gantt Charts: Bar charts showing activities on a timeline, visualizing project schedule, dependencies, and progress.

Section 4: Requirements Engineering

4

4.1 Requirements Gathering and Analysis

Requirements Elicitation: The process of extracting requirements from stakeholders who may have different perspectives and priorities.

Requirements Definition: Formal documentation of what the system must do, including functional and non-functional requirements.

Requirements Specification: Detailed, precise documentation of requirements in a format that developers can use for implementation.

Making Requirements Testable: Requirements must be specific and measurable so that testers can verify whether they have been met.

4.2 Types of Requirements

Functional Requirements: Describe specific behaviors and functions the software must perform. Example: "The system shall calculate payroll."

Non-Functional Requirements: Describe quality attributes and constraints. Examples: performance, security, reliability, usability.

4.3 Requirements Tools and Techniques

Decision Tables: Tabular representations showing how different combinations of input conditions lead to specific outputs.

State Machines: Diagrams showing system states and transitions between states based on events or conditions.

UML State-chart: A standardized notation for representing state machines.

Entity-Relationship Diagram (ERD): Shows entities and their relationships, useful for understanding data requirements.

Class Diagram: Shows classes, attributes, methods, and relationships.

Dataflow Diagram (DFD): Shows how data flows through the system.

Use Case Diagram: Shows actors, use cases, and relationships.

Section 5: System Modeling and Design

5

5.1 Dynamic System Descriptions

Event Traces and Message Sequence Diagrams: Show the sequence of interactions between system components over time.

Petrinets: A formal modeling technique using places (states), transitions (events), and tokens (resources) to model concurrent systems.

5.2 Static System Descriptions

Entity-Relationship Diagram (ERD): Models the data structure of the system, showing entities, attributes, and relationships.

Class Diagram: Shows the object-oriented structure of the system, including classes, inheritance hierarchies, and associations.

Dataflow Diagram (DFD): Models how data flows through the system, showing processes, data stores, and external entities.

Use Case Diagram: Shows system functionality from the user's perspective, including actors and use cases.

5.3 System Descriptions Summary

Dynamic models show behavior over time, static models show structure, and use case models show functionality from the user's perspective. Together, they provide a comprehensive understanding of system requirements and design.

Section 6: Quality Assurance and Testing

6

6.1 Testing Fundamentals

Testing is the process of executing software to identify defects and verify that it meets requirements. Testing includes:

  • Unit Testing: Testing individual components in isolation.
  • Integration Testing: Testing component interactions.
  • System Testing: Testing the complete system.
  • Acceptance Testing: Verifying user requirements are met.

6.2 Quality Metrics

Quality can be measured through various metrics:

Defect Density: Defects per thousand lines of code.

Code Coverage: Percentage of code executed by tests.

Customer Satisfaction: User satisfaction ratings and feedback.

Section 7: Software Maintenance and Evolution

7

7.1 Maintenance Activities

Software maintenance includes multiple types of activities:

Corrective Maintenance: Fixing bugs and defects discovered during operation.

Adaptive Maintenance: Adapting software to new environments or requirements.

Perfective Maintenance: Improving performance or adding new features.

Preventive Maintenance: Making changes to prevent future problems.

7.2 Software Evolution

As software systems age and requirements change, they must evolve. Evolution includes adding new features, improving performance, and adapting to new technologies.

7.3 Legacy Systems

Legacy systems are older software systems that continue to provide value but may be difficult to maintain or modify. Managing legacy systems requires careful planning and incremental modernization.
